# Скрипт очистки базы данных и файловой системы (Bitrix)

![Version](https://img.shields.io/badge/version-0.0.4-blue.svg)
![Python](https://img.shields.io/badge/python-3.6+-green.svg)
![License](https://img.shields.io/badge/license-MIT-lightgrey.svg)

Этот скрипт предназначен для автоматической очистки таблиц в базе данных MySQL и файлов в указанных папках с возможностью последующего копирования файлов.  
Оптимизирован для работы с Bitrix: автоматически считывает настройки подключения из `bitrix/.settings.php`.

## Возможности

- ✅ Очистка таблиц MySQL (TRUNCATE/DROP)
- ✅ Автоматическое извлечение настроек БД из `bitrix/.settings.php`
- ✅ Ручное резервное копирование параметров (fallback в `settings.ini`)
- ✅ Очистка содержимого папок с сохранением исключений
- ✅ Копирование файлов после очистки
- ✅ Резервное копирование (вкл./выкл.)
- ✅ Копирование от имени другого пользователя через `sudo`
- ✅ Подтверждение опасных операций
- ✅ Настройки через INI-файл

## Изменения в версии 0.0.4

- 🚀 **Автоматическое определение корня сайта**  
  Скрипт рекурсивно ищет папку `bitrix`, начиная с первой папки из `clean` или текущей директории.

- 📥 **Приоритетное чтение настроек БД из `bitrix/.settings.php`**  
  Вначале пытается прочитать параметры подключения через `php`. При ошибке — fallback на `settings.ini`.

- 🔁 **Улучшенная логика обработки путей**  
  Все относительные пути интерпретируются относительно корня сайта (например, `preserve_dirs = "bitrix/..."`).

- 🐛 Исправлены ошибки обработки кавычек и пробелов.

## Установка

1. **Установите зависимости:**
```bash
pip3 install pymysql cryptography
```

2. **Создайте файл настроек `settings.ini`:**
```bash
cp settings.ini.example settings.ini
chmod 600 settings.ini  # защищаем файл с паролями
chmod +x cleaner.py
```

3. **Настройте права доступа к системе:**
- Для копирования от имени другого пользователя (`copy_user`) настройте sudo-права:
  ```bash
  # /etc/sudoers.d/cleaner
  your_user ALL=(www-data) NOPASSWD: /bin/cp, /bin/mkdir, /usr/bin/rsync
  ```

## Настройка

Отредактируйте файл `settings.ini`:

### 📊 Секция [database]

| Параметр        | Описание                                          | Пример                  |
|-----------------|---------------------------------------------------|-------------------------|
| `host`          | Хост MySQL                                        | `localhost`             |
| `user`          | Пользователь БД                                   | `your_username`         |
| `password`      | Пароль                                            | `your_password`         |
| `database_name` | Имя БД                                            | `your_database`         |
| `mode`          | Режим работы: `drop_all`, `drop_list`, `truncate` | `drop_all`              |
| `tables`        | Список таблиц для `drop_list`/`truncate`          | `"table1", "table2"`    |
| `auth_plugin`   | Дополнительный плагин аутентификации              | `mysql_native_password` |

> 💡 **ПРИМЕЧАНИЕ**: Если файл `bitrix/.settings.php` найден, скрипт **автоматически извлечет** из него все параметры подключения к БД, включая хост, имя БД, логин и пароль. В этом случае данные в `settings.ini` будут проигнорированы.

Пример:
```ini
[database]
host = "localhost"
user = "your_username"
password = "your_password"
database_name = "your_database"
mode = "drop_all"
tables = "logs", "b_option", "b_lang"
auth_plugin = "mysql_native_password"
```

### 📂 Секция [folders]

| Параметр            | Описание                                            | Пример                                |
|---------------------|-----------------------------------------------------|---------------------------------------|
| `clean`             | Папки для полной очистки содержимого                | `"/tmp/cache", "/var/log/app"`        |
| `copy_sources`      | Папки-источники для копирования                     | `"/path/to/source1"`                  |
| `copy_destinations` | Папки-назначения (должны быть в `clean`)            | `"/tmp/cache"`                        |
| `copy_user`         | Пользователь для выполнения `cp`/`rsync`            | `"www-data"`                          |
| `preserve_dirs`     | Относительные пути к папкам, которые **не удалять** | `"bitrix/php_interface", "upload"`    |
| `preserve_files`    | Относительные пути к файлам, которые **не удалять** | `"bitrix/.settings.php", ".htaccess"` |

> ⚠️ Все пути `preserve_*` задаются **относительно корня сайта**, который определяется автоматически (см. выше).

Пример:
```ini
[folders]
clean = "/tmp/cache", "/var/log/app"
copy_sources = "/path/to/source1", "/path/to/source2"
copy_destinations = "/tmp/cache", "/var/log/app"
copy_user = "www-data"
preserve_dirs = "bitrix/php_interface", "upload", "local/templates"
preserve_files = "bitrix/.settings.php", ".htaccess"
```

### 💾 Секция [backup]

| Параметр     | Описание                                 | Пример          |
|--------------|------------------------------------------|-----------------|
| `enable`     | Вкл/выкл резервное копирование           | `true`          |
| `backup_dir` | Папка для копий (пусто = автоматическая) | `"/tmp/backup"` |

Пример:
```ini
[backup]
enable = true
backup_dir = "/tmp/backup"
```

### 🔐 Секция [security]

| Параметр                         | Описание                                               | Пример |
|----------------------------------|--------------------------------------------------------|--------|
| `confirm_destructive_operations` | Запрашивать подтверждение перед `drop_all`/`drop_list` | `true` |

Пример:
```ini
[security]
confirm_destructive_operations = true
```

## Режимы работы с БД

| Режим       | Описание                                            | Требует `tables` |
|-------------|-----------------------------------------------------|------------------|
| `drop_all`  | Удаляет **все** таблицы в БД                        | ❌                |
| `drop_list` | Удаляет только указанные таблицы                    | ✅                |
| `truncate`  | Очищает данные в указанных таблицах (без структуры) | ✅                |

## Использование

```bash
# Базовый запуск
./cleaner.py

# Сохранение логов
./cleaner.py | tee cleanup_$(date +%Y%m%d_%H%M%S).log
```

## Пример workflow

1. Автоматическое определение корня сайта (`/var/www/html`)
2. Считывание настроек БД из `bitrix/.settings.php` или `settings.ini`
3. Создание резервной копии (если включено)
4. Очистка таблиц БД (drop_all / drop_list / truncate)
5. Сохранение исключений во временную папку
6. Очистка содержимого указанных папок
7. Восстановление исключений из temp-папки
8. Назначение прав на очищенные папки (`chown` на `copy_user`)
9. Копирование файлов из `copy_sources` в `copy_destinations`

## Копирование от другого пользователя

Для копирования с сохранением прав определенного пользователя:

1. Укажите пользователя в `copy_user` (например, `www-data`).
2. Убедитесь, что текущий пользователь имеет права sudo для него.
3. Добавьте в `/etc/sudoers.d/cleaner`:
   ```bash
   your_user ALL=(www-data) NOPASSWD: /bin/cp, /bin/mkdir, /usr/bin/rsync
   ```

## Безопасность

- 🔐 Пароли БД хранятся в `settings.ini` (права `600`)
- ⚠️ Подтверждение для опасных операций (настраивается)
- 📁 Резервные копии создаются в `backup_dir` (или автоматически в `/tmp`)
- 🔒 Все `chown` и `cp` выполняются от имени указанного пользователя (через sudo)

## Логирование

Скрипт выводит подробные логи в консоль. Для сохранения логов:

```bash
./cleaner.py 2>&1 | tee cleanup_$(date +%Y%m%d_%H%M%S).log
```

## Ошибки и устранение неполадок

### ❌ Ошибка подключения к БД
- Проверьте параметры в `settings.ini`
- Убедитесь, что MySQL запущен и доступен
- Проверьте `auth_plugin`: для старых версий MySQL может потребоваться `mysql_native_password`

### ❌ Не найден корень сайта
- Убедитесь, что папка `bitrix` существует в структуре сайта
- Проверьте, что папки из `clean` действительно принадлежат сайту

### ❌ Ошибки `sudo` при копировании
- Проверьте настройки в `/etc/sudoers.d/cleaner`
- Убедитесь, что скрипт запущен от пользователя с правами на `sudo -u www-data`

### ❌ Проблемы с резервным копированием
- Проверьте доступное место на диске (`df -h`)
- Убедитесь, что у пользователя есть права на запись в `backup_dir`

## Важные предупреждения

⚠️ **Режим `drop_all` необратимо удаляет ВСЕ таблицы в базе!**  
⚠️ **Всегда тестируйте скрипт на тестовой среде перед использованием в продакшене**  
⚠️ **Убедитесь, что резервные копии создаются корректно (проверяйте `/tmp/backup`)**  
⚠️ **Для корректной работы `preserve_*` убедитесь, что пути заданы относительно корня сайта**
